# Project notes — Crystallize AI Sidebar

Living design notes for the app. The [README](README.md) is the public-facing entry point; this file captures intent and decisions.

## Vision

A generic, prompt-driven editing surface for Crystallize items. Where the [translation app](https://github.com/CrystallizeAPI/translation-app) does one thing (translate field A → field B), this app lets an editor say "make the description more compelling" or "generate SEO for this page" and have an AI produce structured updates that are applied through the Core API.

## Principles

1. **Stay inside Crystallize's shape.** The AI never invents component IDs or types — it receives the exact shape and returns updates keyed to real paths.
2. **Nested content is first class.** Pieces and chunks are flattened into addressable paths (`meta.title`, `specs[0].width`) so the AI and the UI both treat them as ordinary fields.
3. **Preview, then apply.** Every AI response renders as a diff; the editor decides whether to apply each update.
4. **Component-aware output.** The system prompt tells the model what kind of field it's writing (meta title vs. headline vs. body), including length targets and anti-duplication rules.
5. **Refresh, don't reload.** After a successful write, we call `signal.send("refetchItemComponents", …)` so the editor updates in place.

## Data model

`lib/types.ts` extends Crystallize component content with:

- `ComponentContent` — adds `identifier`, nested `components`, and `chunks`
- `ShapeComponent` — adds optional `config` (so we can inspect piece/chunk shapes)

`lib/utils.ts#flattenEditableComponents` walks the tree and returns `EditableLeaf[]`:

```ts
type EditableLeaf = {
  path: string;            // "title" | "meta.title" | "specs[0].width"
  label: string;
  parentLabel?: string;
  type: ComponentType;
  content?: ComponentContent;
};
```

Everything else (AI context, UI chips, preview, update) operates on `path`.

## Update pipeline

`POST /api/update` handles both flat and nested paths:

- Flat path (`"title"`) → direct `ComponentInput` mutation.
- Piece path (`"meta.title"`) → re-fetch the piece, build a full `PieceContentInput` with all siblings preserved, replace just the targeted leaf.
- Chunk path (`"specs[0].width"`) → re-fetch the chunk row, rebuild all chunk rows, replace just the targeted leaf.

`componentContentToInputPayload` converts each content kind back to the input shape Crystallize expects (e.g. stripping read-only fields from rich text JSON).

## AI prompt

`lib/ai.ts` builds the system prompt with:

- Shape identifier, item name, item type, language
- The list of editable leaves the user has selected, formatted as `- path — ParentLabel / Label [type]`
- The current value of each leaf (string, JSON, or paragraph summary)
- Known image keys (so the model re-uses real media instead of hallucinating)

And a rules block covering:

- JSON-only output format
- Only emit updates for paths the user explicitly targets
- Rich text must use Crystallize's block/inline JSON (never HTML)
- Component-aware length targets (meta title 50–60, meta description 140–160, OG title 40–70, OG description ≤200, editorial title 40–80, intro 1–3 sentences, slug kebab-case <60)
- A meta title must never be identical to the page title
- Do not reuse the same text across different fields

## UI surface

- **ItemContext** — name, type, shape, language
- **AIPrompt** — prompt textarea, from/to language selectors, component selection chips ("All" / individual leaves / "Clear")
- **UpdatePreview** — one card per proposed update. Resolves `path` against the live item, renders before/after, shows an "Apply" per card
- **RichTextPreview** — walks the rich-text JSON AST and renders with Crystallize design-system styling
- **ImageUpload** — upload to Crystallize media library and pick from existing images

## Known limitations

- No streaming of AI output yet — previews appear once the full response arrives.
- No prompt history or undo. The editor can always revert via Crystallize's version history.
- Images inside pieces can be rendered but not yet regenerated by the AI (alt text only).
- Product variants are supported for reading but not yet for writing.
